Portfolio project page anatomy — what sections and elements to include

Must have
Highly recommended
Nice to have
The hero — first thing they see
Hero screenshot or demo Must have

A big, high-quality screenshot or short video/GIF showing the app in action. This is the first impression — make it count. Show the most visually impressive or useful part of the UI.

Project title + one-liner Must have

A clear name and a single sentence explaining what it does and who it's for. E.g. "TaskFlow — a drag-and-drop task manager for remote teams." No jargon, no buzzwords.

Tech stack tags Must have

Small pill badges showing the core technologies used — React, PostgreSQL, AWS, etc. Recruiters scan these instantly. Keep it to 4–6 most relevant ones.

The story — what you built and why
Problem + solution writeup Must have

A short narrative — what problem were you solving? What decisions did you make and why? This shows how you think, not just what you built. 2–4 paragraphs is ideal.

Key features list Must have

3–6 bullet points of the most interesting or technically challenging things the app does. Focus on things that demonstrate your skill, not generic stuff like "user authentication."

Multiple screenshots / screen recording Highly recommended

Show different states of the app — a dashboard, a detail view, a mobile view, an empty state, an error state. 3–5 images tells a much fuller story than one hero shot.

The proof — show it's real and working
GitHub link Must have

Link directly to the repo. Make sure the README is clean — it's the first thing interviewers open. Include setup instructions, a brief overview, and screenshots inside the README too.

Live demo link Highly recommended

A deployed version they can actually click around. Vercel, Netlify, Railway — pick one. If the project can't be live (cost, private data), a video walkthrough is the next best thing.

Demo video / Loom walkthrough Highly recommended

A 2–3 minute walkthrough where you show and explain the app. This is especially powerful if the live demo is unavailable or hard to explore without context. Embed it directly on the page.

The depth — for technical credibility
Technical challenges & decisions Highly recommended

What was hard? What trade-offs did you make? ("I chose Redis for session caching over cookies because…") This is gold for senior roles — shows you can reason about architecture, not just implement it.

Architecture diagram Nice to have

A simple diagram of how the system fits together — frontend, backend, database, APIs, third-party services. Very impressive for full-stack or backend-heavy projects.

Metrics or outcomes Nice to have

If the project was used by real people, mention it. "Used by 200+ beta users", "reduced load time by 60%", "deployed at company X." Numbers build instant credibility.

What you'd do next / known limitations Nice to have

Shows self-awareness and product thinking. "If I had more time I'd add X" or "the current caching strategy doesn't scale to Y" tells interviewers you understand your own work critically.

Navigation — don't trap them
Back to portfolio / next project Must have

Easy navigation back to the main projects list and ideally a "next project →" link. Never make a recruiter hit the browser back button to see more of your work.
